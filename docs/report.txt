CHAPTER 1: INTRODUCTION
1.1 GENERAL OVERVIEW
In today's digital world, businesses are rapidly adopting computerized systems to manage their daily operations efficiently. However, many small and medium-sized businesses still rely on traditional methods such as handwritten bills, registers, or spreadsheet software for managing billing and inventory records. These methods are time-consuming, difficult to maintain, and often result in human errors, incorrect calculations, stock mismatches, and inefficient record management.
The "Billing and Inventory Management System" is a web-based business management application developed to solve these problems by automating billing and inventory-related operations. The main purpose of this project is to provide a simple, secure, and efficient platform that helps businesses manage products, customers, invoices, inventory, and reports from a centralized system.
The system is developed using FastAPI for the backend, with a frontend powered by HTML, CSS, and JavaScript. Data persistence is managed via an SQLite database. The application architecture maintains a clear separation between business logic, the presentation layer, and database operations, ensuring a modular and maintainable codebase.
The system allows users to perform various business operations such as product management, customer management, invoice generation, payment tracking, inventory monitoring, report generation, and administrative management. The billing process becomes faster and more accurate because the system automatically calculates totals, taxes, and updates stock quantities after every transaction.
One of the important features of this system is that it maintains a proper relationship between billing and inventory modules. Whenever an invoice is generated, the corresponding stock quantity is automatically reduced, which helps maintain accurate inventory records. The system also provides reporting features that help business owners analyze their sales and business performance.
From an academic perspective, this project provides practical experience in software development, database design, web technologies, authentication mechanisms, and business process automation. The project combines various concepts studied during the Bachelor of Computer Applications program and demonstrates their implementation in a real-world business application.

1.2 RATIONALE
The idea of developing the Billing and Inventory Management System came from observing the problems faced by small businesses while managing their daily operations. Many local shops, retailers, wholesalers, and service providers still depend on manual record keeping methods. These methods create several challenges such as maintaining customer records, calculating bills manually, managing stock quantities, and generating reports.
Manual billing systems require significant time and effort. Even a small mistake in calculation can affect business records and financial reports. Similarly, inventory management becomes difficult when stock updates are performed manually, which may lead to overstocking or stock shortages.
Although many commercial billing software applications are available in the market, they are often expensive and contain features that may not be necessary for small businesses. Some software also requires specialized training and technical knowledge.
The main objective behind developing this project was to create a simple, affordable, and user-friendly billing solution that can automate business operations while remaining easy to understand and operate. This system helps reduce manual effort, improves accuracy, and provides better control over business activities.
In addition to solving a practical business problem, this project also provides an opportunity to apply programming concepts, database management techniques, software engineering principles, and web development technologies learned during the BCA course.

Before the implementation of computerized billing systems, businesses traditionally relied on manual methods for maintaining sales and inventory records. Even today, many small businesses continue to use these methods due to lack of technical knowledge or financial resources.

Manual Billing System
In manual billing systems, all transactions are recorded using registers, notebooks, or printed bills. Inventory records are also maintained separately by employees. This method has several disadvantages:
High chances of human error

Spreadsheet-Based System

Various standalone billing software applications are available commercially. These systems provide billing functionality but often suffer from the following drawbacks:

Vendor dependency
Additional maintenance costs
Due to these limitations, there is a need for an integrated system that can efficiently manage billing, inventory, customers, products, and business reports through a single platform.

1.8 LIMITATIONS
Although the Billing and Inventory Management System provides many useful features for managing business operations, there are still some limitations in the current version of the project.
One of the major limitations is that the system is developed as a web application and currently does not provide a dedicated mobile application. Users need internet access and a web browser to access the system.
The project utilizes SQLite for data management, which provides an efficient, serverless solution for small to medium-sized enterprises. While suitable for the current scope, very large organizations with massive concurrent transactions may require migration to a client-server database in future iterations.
The current version of the project also does not support barcode scanning and barcode printing features. Product entries and invoice creation are performed manually through the user interface.
Another limitation is that the system does not currently provide direct integration with GST filing portals, accounting software, or third-party payment gateways. These features can be implemented in future versions.
The project also supports a limited number of user roles and does not include advanced enterprise-level features such as multi-branch management, warehouse management, supplier management, or cloud synchronization.
Despite these limitations, the current system successfully fulfills the primary objective of automating billing and inventory operations for small and medium businesses.

4.3 DATA INTEGRITY AND CONSTRAINTS
Data integrity is an important aspect of database design because it ensures that the information stored in the database remains accurate, consistent, and reliable. The Billing and Inventory Management System uses various database constraints and validation mechanisms to maintain data integrity. The following constraints have been implemented in the system:

Primary Key Constraints
Every table contains a primary key that uniquely identifies each record. Examples include:
- id in the users table
- id in the categories table
- id in the products table
- id in the customers table
- id in the invoices table
- id in the invoice_items table
- id in the payments table
- id in the logs table
- key in the settings table (text-based primary key)
- id in the inventory table
- id in the stock_transactions table
Primary keys prevent duplicate records and ensure uniqueness of each row.

Foreign Key Constraints
Foreign keys are used to establish relationships between different tables and maintain referential integrity:
- category_id in the products table references id in the categories table.
- customer_id and user_id in the invoices table reference id in the customers table and id in the users table, respectively.
- invoice_id and product_id in the invoice_items table reference id in the invoices table and id in the products table, respectively.
- invoice_id in the payments table references id in the invoices table.
- product_id in the inventory table references id in the products table.
- product_id and user_id in the stock_transactions table reference id in the products table and id in the users table, respectively.
SQLite foreign key enforcement is enabled programmatically on every connection handle via "PRAGMA foreign_keys = ON;".

Unique Constraints
Certain fields require unique values to avoid duplicate data:
- username in the users table.
- name in the categories table.
- sku in the products table.
- phone in the customers table.
- invoice_number in the invoices table.
- product_id in the inventory table.
- key in the settings table.

Not Null Constraints
Mandatory fields are marked as NOT NULL to prevent incomplete data:
- username, fullname, password_salt, password_hash, and role in the users table.
- name in the categories table.
- name, sku, price, stock, min_stock, and gst_rate in the products table.
- name and phone in the customers table.
- invoice_number, subtotal, total, and payment_status in the invoices table.
- invoice_id, quantity, unit_price, and total_price in the invoice_items table.
- invoice_id, amount, and payment_method in the payments table.
- action in the logs table.
- value in the settings table.

Validation Constraints
Business rules are validated both at the Pydantic schema layer on the server and at the database constraint level:
- Product price must be greater than zero.
- Product stock and min_stock must be greater than or equal to zero.
- Product GST rate must be greater than or equal to zero.
- Customer phone number must be exactly 10 digits.
- Invoice subtotal, gst, discount, and total must be greater than or equal to zero.
- Invoice items quantity must be greater than zero.
- Payment amount must be greater than zero.
- User role must be restricted to 'admin' or 'staff'.
- Product location and inventory location must be restricted to 'storefront' or 'warehouse'.
- Payment method must be restricted to 'cash', 'card', 'upi', or 'credit' for invoices, and 'cash', 'card', or 'upi' for payment collections.
- Stock transaction type must be restricted to 'restock', 'sale', or 'adjustment'.

Data Integrity Measures
To improve data consistency, the system implements:
- Client-side input validation via HTML5 attributes and JavaScript.
- Server-side validation using Pydantic schemas.
- Database transaction blocks to ensure atomic checkout operations.
- Detailed activity and audit logging.
- Parameterized SQL queries via the sqlite3 library to prevent SQL injection.
- Database cascading deletes on dependent tables (e.g. deleting an invoice deletes its items and payments).



4.4 DATABASE DESIGN
The Billing and Inventory Management System uses a relational database design to manage business data efficiently. The database has been normalized to reduce redundancy and improve data consistency. The database consists of the following 11 tables and triggers:

1. Users Table (users)
Stores the credentials and roles of application users.
- id: Primary Key, unique user identifier.
- username: Unique alphanumeric login name.
- password_salt: Random salt generated during account creation.
- password_hash: PBKDF2 HMAC SHA-256 digest of password.
- password_plaintext: Stored plaintext password (strictly for viva defense/testing verification).
- fullname: User's full display name.
- role: Authorization level ('admin' or 'staff').
- created_at: Timestamp of account creation.

2. Categories Table (categories)
Stores product categories to classify products.
- id: Primary Key, unique category identifier.
- name: Unique category name (e.g., Electronics, Groceries).
- description: Text details about the category.

3. Products Table (products)
Maintains the catalog of products sold in the store.
- id: Primary Key, unique product identifier.
- sku: Unique Stock Keeping Unit for barcode scanning / code identifier.
- category_id: Foreign key referencing categories(id).
- name: Product display name.
- price: Unit price of product (excluding tax).
- stock: Current stock quantity (synchronized with inventory.stock).
- min_stock: Minimum stock alert threshold level.
- gst_rate: Standard tax rate applied to product (e.g., 18.0%).
- location: Current physical location of stock ('storefront' or 'warehouse').
- description: Details or specifications.
- created_at: Product entry creation timestamp.

4. Customers Table (customers)
Stores CRM customer profiles.
- id: Primary Key, unique customer identifier.
- name: Customer name.
- phone: Unique 10-digit customer mobile number.
- email: Customer email address.
- address: Customer physical shipping address.
- created_at: Customer registration timestamp.

5. Invoices Table (invoices)
Represents billing transactions.
- id: Primary Key, unique invoice transaction ID.
- invoice_number: Formatted unique invoice number (e.g., INV-20260706-0001).
- customer_id: Foreign key referencing customers(id) (nullable for guest checkout).
- user_id: Foreign key referencing users(id) of the cashier staff.
- subtotal: Cumulative pre-tax total of all items.
- discount: Flat discount amount deducted from subtotal.
- gst: Calculated tax amount (CGST + SGST).
- total: Net grand total of the transaction.
- payment_status: Billing state ('paid', 'pending', or 'unpaid').
- created_at: Transaction execution date.

6. Invoice Items Table (invoice_items)
Line items of individual sales invoices.
- id: Primary Key.
- invoice_id: Foreign key referencing invoices(id) with Cascade Delete.
- product_id: Foreign key referencing products(id) with Set Null.
- product_name: Snapshot of product name at transaction time.
- product_sku: Snapshot of product SKU at transaction time.
- quantity: Number of units purchased.
- unit_price: Price per unit at transaction time.
- total_price: Line item subtotal before tax.
- gst_rate: Percentage tax rate applied to this specific line item.

7. Payments Table (payments)
Tracks transaction payment history.
- id: Primary Key.
- invoice_id: Foreign key referencing invoices(id) with Cascade Delete.
- amount: Numeric amount collected.
- payment_method: Payment option chosen ('cash', 'card', or 'upi').
- transaction_reference: Bank or payment reference numbers.
- created_at: Payment logging date.

8. Activity Logs Table (logs)
Internal system audit ledger.
- id: Primary Key.
- user_id: Foreign key referencing users(id) who executed the action.
- action: String event code (e.g., 'LOGIN', 'PRODUCT_CREATE').
- details: Human readable details about the audit event.
- created_at: Audit event timestamp.

9. System Settings Table (settings)
System-wide configuration flags.
- key: Unique setting parameter name (e.g., company_name, default_gst).
- value: Configuration value string.

10. Inventory Table (inventory)
Advanced inventory control tracking real-time stock levels.
- id: Primary Key.
- product_id: Unique foreign key referencing products(id) with Cascade Delete.
- stock: Current stock count available.
- min_stock: Minimum stock threshold.
- location: Current physical location of inventory ('storefront' or 'warehouse').
- last_updated: Timestamp of the most recent stock alteration.

11. Stock Transactions Table (stock_transactions)
Detailed stock flow ledger logging all entries and exits.
- id: Primary Key.
- product_id: Foreign key referencing products(id) with Cascade Delete.
- quantity: Difference quantity (+ve for restocks, -ve for sales/adjustments).
- transaction_type: Event category ('restock', 'sale', 'adjustment').
- user_id: Foreign key referencing users(id) of staff performing the operation.
- created_at: Transaction timestamp.
- notes: Audit remarks (e.g., "Initial stock import", "Invoice INV-20260706-0001").

Database Triggers
The SQLite database implements two automatic execution triggers to enforce business data synchronization:
- auto_create_inventory_row: Triggered AFTER INSERT ON products. It automatically creates a corresponding inventory tracking row in the inventory table with the product's default stock, min_stock, and location details.
- sync_product_stock_after_update: Triggered AFTER UPDATE OF stock, min_stock, location ON inventory. It automatically synchronizes and propagates the changes back to the products table to maintain denormalized data consistency.

4.8 CORE BILLING CALCULATION MODEL
The Billing and Inventory Management System performs all financial and tax calculations on the server side using the following formulas:

1. Pre-Tax Line Subtotal
For each product item selected in the invoice:
Line Subtotal = Quantity * Unit Price

2. Cumulative Subtotal
The sum of all pre-tax line subtotals:
Subtotal = Sum of all Line Subtotals

3. GST Tax Calculation
Tax is calculated on the pre-tax line subtotal based on the product's assigned GST rate:
Line GST = Line Subtotal * (GST Rate / 100)
Total GST = Sum of all Line GSTs

For financial and tax reporting, the total GST is split equally into Central GST (CGST) and State GST (SGST) per line item:
CGST = Line GST / 2
SGST = Line GST / 2

4. Discount Capping
To prevent a negative grand total, the requested flat discount is capped against the sum of the subtotal and total GST:
Actual Discount = Minimum(Requested Discount, Subtotal + Total GST)

5. Grand Total Calculation
The final net payable amount is computed by:
Grand Total = Subtotal + Total GST - Actual Discount

CHAPTER 6: SYSTEM SECURITY MEASURES
Security is a fundamental requirement for the Billing and Inventory Management System. Multiple security layers have been implemented to protect business data, ensure user privacy, and maintain system integrity.
6.1 USER AUTHENTICATION
The system provides a secure login mechanism to verify user identity. Users must provide valid credentials (username and password) to access the system modules.
6.2 JWT TOKEN AUTHENTICATION
The application utilizes JSON Web Tokens (JWT) for secure communication between the frontend and backend. Tokens are generated upon successful login and are required for all subsequent API requests.
6.3 PASSWORD SECURITY
Passwords are never stored in plain text. The system uses the PBKDF2 HMAC SHA-256 password hashing algorithm (with 120,000 iterations and a cryptographically secure random salt) to hash passwords before storing them in the database.
6.4 ROLE-BASED ACCESS CONTROL (RBAC)
Access to various modules is restricted based on user roles (Admin, Staff). Each role has specific permissions to ensure that users can only perform authorized actions.
6.5 SESSION MANAGEMENT
Secure session management ensures that user sessions are handled safely. Sessions are designed to expire after a period of inactivity to prevent unauthorized access.
6.6 INPUT VALIDATION
All user inputs are validated on the server side to ensure they meet the expected format, type, and length, preventing malicious data entry.
6.7 SQL INJECTION PREVENTION
The system uses parameterized queries via the standard sqlite3 library to interact with the database, effectively neutralizing SQL injection threats.
6.8 API SECURITY
REST APIs are protected using authentication headers to ensure that only authorized clients can access system services.
6.9 ACTIVITY LOGGING
The system maintains detailed logs of user activities, including login attempts, data modifications, and sensitive transactions, for auditing and monitoring.
6.10 ERROR HANDLING AND EXCEPTION MANAGEMENT
Robust error handling mechanisms prevent the disclosure of sensitive system information in error messages while ensuring system stability.
6.11 BACKUP AND DATA PROTECTION
Database backups are managed manually by copying the single-file SQLite database. Data integrity constraints are also implemented within the database schema to ensure relational security.
6.12 SYSTEM SECURITY ARCHITECTURE
The overall architecture follows standard practices for security, including secure client-side token storage and server-side request authorization.

    )

4. Process Invoice Creation & Stock Deductions
```python
@router.post("/api/invoices")
def create_invoice(payload: InvoiceCreate, current_user: dict[str, Any] = Depends(get_current_user)) -> dict[str, Any]:
    invoice_number = generate_invoice_number()

    if payload.payment_method not in ("cash", "card", "upi", "credit"):
        raise HTTPException(status_code=400, detail="Invalid payment method")

    with connect_db() as conn:
        subtotal = 0.0
        total_gst = 0.0
        items_data = []

        for item in payload.items:
            product_row = conn.execute(
                """
                SELECT p.id, p.name, p.sku, p.price, i.stock, i.min_stock, i.location
                FROM products p
                LEFT JOIN inventory i ON i.product_id = p.id
                WHERE p.id = ?
                """,
                (item.product_id,)
            ).fetchone()
            if not product_row:
                raise HTTPException(status_code=400, detail=f"Product ID {item.product_id} not found")
            product = dict(product_row)

            if product["stock"] < item.quantity:
                raise HTTPException(
                    status_code=400,
                    detail=f"Insufficient stock for '{product['name']}'. Available: {product['stock']}, Requested: {item.quantity}",
                )

            item_total = item.unit_price * item.quantity
            item_gst = item_total * (item.gst_rate / 100.0)
            subtotal += item_total
            total_gst += item_gst

            items_data.append({
                "product_id": item.product_id,
                "product_name": product["name"],
                "product_sku": product["sku"],
                "quantity": item.quantity,
                "unit_price": item.unit_price,
                "total_price": item_total,
                "gst_rate": item.gst_rate,
            })

        discount = min(payload.discount, round(subtotal + total_gst, 2))
        grand_total = round(subtotal + total_gst - discount, 2)
        
        is_credit = (payload.payment_method == "credit")
        payment_status = "unpaid" if is_credit else "paid"

        invoice_id = execute_write_conn(
            conn,
            """INSERT INTO invoices (invoice_number, customer_id, user_id, subtotal, discount, gst, total, payment_status)
               VALUES (?, ?, ?, ?, ?, ?, ?, ?)""",
            (invoice_number, payload.customer_id, current_user["id"], round(subtotal, 2), round(discount, 2), round(total_gst, 2), grand_total, payment_status),
        )

        for item_data in items_data:
            execute_write_conn(
                conn,
                """INSERT INTO invoice_items (invoice_id, product_id, product_name, product_sku, quantity, unit_price, total_price, gst_rate)
                   VALUES (?, ?, ?, ?, ?, ?, ?, ?)""",
                (invoice_id, item_data["product_id"], item_data["product_name"], item_data["product_sku"],
                 item_data["quantity"], item_data["unit_price"], item_data["total_price"], item_data["gst_rate"]),
            )
            
            conn.execute(
                "UPDATE inventory SET stock = stock - ?, last_updated = CURRENT_TIMESTAMP WHERE product_id = ?",
                (item_data["quantity"], item_data["product_id"]),
            )

            conn.execute(
                """INSERT INTO stock_transactions (product_id, quantity, transaction_type, user_id, notes)
                   VALUES (?, ?, 'sale', ?, ?)""",
                (item_data["product_id"], -item_data["quantity"], current_user["id"], f"Invoice {invoice_number}")
            )

        if not is_credit:
            execute_write_conn(
                conn,
                "INSERT INTO payments (invoice_id, amount, payment_method, transaction_reference) VALUES (?, ?, ?, ?)",
                (invoice_id, grand_total, payload.payment_method, payload.transaction_reference or ""),
            )

        conn.commit()

    add_log(current_user["id"], "INVOICE_CREATE", f"Generated invoice {invoice_number} — ₹{grand_total} ({payment_status.upper()})")
    return {"message": "Invoice created", "invoice_number": invoice_number, "id": invoice_id, "total": grand_total}
```
